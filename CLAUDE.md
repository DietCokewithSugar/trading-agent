# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

AI 新闻交易员 — a news-driven US stock paper-trading site. It fetches financial news (FMP API + Yahoo RSS), has DeepSeek classify each article as bullish/bearish on a four-tier scale, lets DeepSeek decide simulated buy/sell orders under server-side risk constraints, and shows the portfolio live in a React dashboard. Everything is virtual money; there is no real brokerage integration.

## Commands

```bash
npm install                # root deps (Express server)
npm run build              # cd web && npm install && vite build → web/dist
npm start                  # node server/index.js → http://localhost:3000 (serves API + built frontend)
npm test                   # node --test → test/*.test.js (pure-function unit tests, no env/network needed)

# Frontend dev with hot reload (two terminals):
npm start                  # terminal 1: backend on :3000
cd web && npm run dev      # terminal 2: Vite on :5173, /api proxied to :3000
```

- Requires Node >= 22. Unit tests cover the pure functions (market calendar, slippage model, tier/credibility/final-confidence math, sizing chain, eligibility gate, forward returns, signal stats); CI (`.github/workflows/ci.yml`) runs `npm test` + the frontend build on every push/PR. No linter configured. When adding pure logic, prefer extracting it into a testable module (see `sizing.js`, `eligibility.js`) and add a test.
- `.env` (copy from `.env.example`) needs `FMP_API_KEY`, `DEEPSEEK_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`. The server still boots without them (API + static hosting work) but the scheduler does not start. All tunables live in `server/config.js`; document new env vars in `.env.example`, `README.md`, and `render.yaml`.
- Database: run `supabase/schema.sql` in the Supabase SQL Editor for a fresh install; existing deployments apply incremental scripts from `supabase/migrations/` instead.
- Deployed on Render via `render.yaml` (build = `npm install && npm run build`, start = `npm start`, health check = `/api/health`).
- Manual trigger for a full cycle: `POST /api/run-cycle` (requires `x-admin-token` header if `ADMIN_TOKEN` is set; anonymous triggers share a global 120s cooldown and get 429 inside it). The endpoint exists for external cron services — the public dashboard has no trigger button; in-site manual runs go through `#/admin`.
- Admin surface: the hidden frontend route `#/admin` plus `/api/admin/*` (`server/routes/admin.js`) — verify/status/run-cycle/reset. These endpoints **require** `ADMIN_TOKEN` (503 when unset, 403 on mismatch); token comparison is constant-time (`authGuard.js#safeTokenEqual`) and failed attempts are rate-limited per IP (10 per 15min via `express-rate-limit`, successes don't count) on both `/api/admin/*` and `/api/run-cycle`. `POST /api/admin/reset` wipes all business data and restores cash to `INITIAL_CAPITAL`: it sets the global halt flag (`server/services/halt.js`), drains the running cycle, then truncates via the `admin_reset_data` RPC (005 migration; per-table delete fallback) inside the trade lock, clears in-process caches, and broadcasts a `reset` SSE event.

## Architecture

Single Node process: Express serves `/api/*` and the built frontend from `web/dist`, and `server/scheduler.js` runs seven `setInterval` loops (news poll, quote push, snapshot, risk check, daily position review, open-order queue, signal forward-return backfill). There is no queue or worker — all background work happens in-process, guarded by simple re-entrancy flags (`cycleStatus.running`, `running` in riskMonitor/positionReview/openQueue/signalReturns) plus the global halt flag in `halt.js` (set during admin reset; runCycle, checkStops, processPendingOrders and backfillForwardReturns early-return while halted).

### The trading pipeline (core flow)

`runCycle()` in `server/services/newsService.js` orchestrates one round and is the place to start reading:

1. **Fetch** — `fmp.js` (stock/general/press news) + `yahoo.js` (RSS for watchlist + held symbols). Fast polls (every `NEWS_POLL_SECONDS`, default 20s) only fetch stock news; roughly every 5 minutes a `fullFetch` round adds the other sources. Articles are upserted into `news_articles` with `onConflict: 'url', ignoreDuplicates: true`, so only genuinely new rows flow onward. At normalization time each article gets a **source credibility score** (`credibility.js#scoreSource`, stored in `source_domain`/`source_score`, 009 migration): the *original* source is scored by article-URL domain (Reuters/AP/SEC 0.95 → press wires 0.90 → Bloomberg/WSJ/CNBC/Yahoo 0.85 → opinion platforms 0.65 → low-trust sites 0.50 → unknown 0.40), minus a 0.03 aggregation penalty for FMP-relayed articles — FMP/Yahoo are discovery channels, not the scored source.
2. **Analyze** — candidates come from a backlog query (articles from the last 24h with `analyzed_at IS NULL`), not just the current round's inserts, capped at `MAX_ANALYZE_PER_CYCLE` per round; `analyzed_at` is stamped after each analysis (including irrelevant verdicts) so failures retry next round. `deepseek.js#analyzeArticle` returns sentiment, impact strength/scope, confidence, and a one-line `event_summary`. Tier is computed in code by `computeTier` (1 = strong+wide … 4 = weak+narrow), not by the LLM. `analyzeAndStore` also computes `final_confidence` = source_score × LLM confidence × recency (1.0 within 1h, floor 0.5 at 24h+) × tier materiality (`credibility.js#computeFinalConfidence`). Every non-neutral tiered analysis also gets its **signal-time market price** stamped (`signalReturns.js#recordSignalPrice` → `news_analyses.signal_price`, 011 migration) — including signals later deduped/held/vetoed — feeding the evaluation layer below. A signal is actionable only if non-neutral, `tier <= TRADE_TIER_THRESHOLD` (default 2), and confidence ≥ 0.5.
3. **Dedup + cross-source confirmation** — `eventService.js#resolveEvent`: DeepSeek compares the event summary against the symbol's `news_events` rows from the last `EVENT_DEDUP_HOURS` (72h); duplicate coverage of the same underlying event only bumps `article_count` and never trades again. At the threshold comparison, bullish signals from press-release sources (`credibility.js#isPressRelease`: `fmp-press` or newswire domains) are discounted by `PRESS_BULLISH_PENALTY` (default ×0.75) — press releases are factual but inherently promotional and the classic micro-cap pump channel, so a lone PR usually lands in the held bucket until an independent outlet confirms. Signals whose (penalized) `final_confidence < MIN_FINAL_CONFIDENCE` (default 0.35) are **held**: the event is recorded but not traded. If a later report of a still-untraded event arrives from an *independent* domain (tracked in `news_events.source_domains`), `resolveEvent` returns `confirmable: true` and `runCycle` re-evaluates with the new article's own final_confidence × confirmation boost (+10% per extra source, cap ×1.2) — a lone low-trust site can't trigger a trade, but a Reuters/Yahoo follow-up confirms it. New/confirmed events still pass a same-direction trade cooldown (`TRADE_COOLDOWN_MINUTES`, 30min) on the `trades` table. **Fail-closed convention:** if dedup checks error out, the trade is skipped — prefer missing a trade over double-trading (likewise, no cross-confirmation when `source_domains` is missing).
4. **Trade** — `trader.js#handleSignal` fetches quote + company profile. Bullish signals must first pass the **eligibility gate** (`eligibility.js#checkBuyEligibility`, before any LLM call): min market cap (`MIN_MARKET_CAP`, $300M), min share price (`MIN_PRICE`, $2), min average dollar volume (`MIN_AVG_DOLLAR_VOLUME`, $5M/day); missing profile data fails closed. Then it retrieves past-trade lessons via `memoryService.js#getMemories`, and `deepseek.js#decideTrade` must first pass *symbol validation* (is the news subject really this listed company? do price/market-cap magnitudes match the story?) before choosing buy/sell/hold. This guards against mapping pre-IPO companies to similar tickers (e.g. SpaceX ≠ SPCE). Validation failure forces hold; `isActivelyTrading === false` profiles are skipped before the LLM is even asked. For buys, the sizing chain is (in order): LLM `fraction` (**a share of total portfolio value**, capped by available cash — not a share of remaining cash, which would make position size depend on news arrival order) → tier/confidence/source-credibility multipliers (`sizing.js#scaleFraction`: `config.tierSizeMultipliers`, confidence 0.5–1.0 maps to ×0.5–×1.0, source score maps to ×0.6–×1.0) → risk-officer `scale` → hard caps. The **risk officer** (`deepseek.js#reviewProposedTrade`, gated by `ENABLE_RISK_OFFICER`) reviews each surviving buy with portfolio-level context (position/sector weights, last 5 sell PnLs, memories) and can approve/downsize/veto; only an explicit `approve === true` passes (missing/malformed fields veto), and if its call throws, the buy is skipped (fail-closed). Hard risk caps are enforced in code, not by the LLM: single position ≤ 25% of portfolio, single buy ≤ 20% of total value, min order $50, long-only. Buys store AI-chosen `stop_loss`/`take_profit` prices on the position (clamped to 3–15% / 5–30%). Order execution is serialized through an in-process trade lock (`withTradeLock` in `trader.js`) and, because the LLM decision can take tens of seconds, re-fetches **both** the valuation and a fresh quote at order time: buys abort if the fresh price drifted more than `BUY_PRICE_DRIFT_ABORT_PERCENT` (default 5%) from the decision-time price, then fill through the slippage model in `execution.js#computeFill` (adverse fill: market-cap-tiered half-spread × session × volatility multipliers + size impact + optional `COMMISSION_BPS`, capped at `SLIPPAGE_MAX_BPS`; `ENABLE_SLIPPAGE` to disable); sells fall back to the decision-time reference price if the re-fetch fails (stop orders must execute). The market reference price and applied slippage are recorded on `trades.quote_price`/`slippage_bps` (008 migration, best-effort post-insert update). Orders land atomically via the `execute_trade` Postgres function (cash delta + position upsert + trade insert in one transaction, with the legacy non-transactional path as fallback when the 004 migration is missing). **Open-order queue (010 migration):** when the market session is `closed` at decision time, buys and sells are NOT filled at the stale closing price — `handleSignal` enqueues them into `pending_orders` (returning `{ queued: true }`; runCycle still marks the event consumed) and `openQueue.js#processPendingOrders` (scheduler loop, regular session only) fills them at the day's **opening price** (`quote.open`, falling back to the live price) through `executeQueuedBuy`/`executeSellOrder` — no drift abort there, the overnight gap is intentionally borne by the strategy. Orders expire after `PENDING_ORDER_MAX_AGE_HOURS` (96h); cap/min-amount rejections cancel the order, transient quote failures retry next tick. If the table is missing, enqueue returns null and the trader falls back to the legacy immediate fill. Pre/post sessions still fill immediately at real aftermarket prices.

Separately, `riskMonitor.js#checkStops` (every `RISK_CHECK_SECONDS`, skipped while market is closed) sells the full position when price crosses stop-loss/take-profit, reusing `trader.js#executeSellOrder` with `trigger: 'stop_loss' | 'take_profit'` (news trades use `trigger: 'news'`, daily review sells use `trigger: 'review'`). It also ratchets trailing stops (`ENABLE_TRAILING_STOP`, needs the 007 `peak_price` column): on a new post-entry high, `stop_loss` moves up to `peak × (1 − original stop distance)`, never down.

### Signal evaluation layer (forward returns)

Independent of trading, `signalReturns.js` answers "does the classification itself have alpha": every non-neutral analysis gets `signal_price` at observation time, and a 10-minute scheduler loop backfills `fwd_return_1h` (live quote in the t+60–120min window; left null if missed) and `fwd_return_1d`/`fwd_return_5d` (close of the 1st/5th trading day after the signal's ET date, via daily history — `computeDailyForwardReturns` is pure and tested). `signalStats.js#getSignalStats` (`/api/signal-stats`, "信号质量" tab) aggregates direction-adjusted hit rates / mean returns bucketed by tier, source-score band, confidence (calibration), and traded-vs-untraded, plus a final_confidence↔return Pearson IC per horizon. All of it is migration-tolerant (011) and never touches the trading path.

Two post-trade LLM features (both cheap by design):
- **Reflection/memory** (`memoryService.js`, `ENABLE_REFLECTION`, 006 migration): after any sell with `realized_pnl`, `executeSellOrder` fires `reflectOnClosedTrade` outside the trade lock (fire-and-forget) — DeepSeek distills a transferable lesson into `trade_reflections`; `getMemories` injects ≤5 lessons into later `decideTrade`/risk-officer calls.
- **Daily position review** (`positionReview.js`, `ENABLE_POSITION_REVIEW`): once per trading day after `POSITION_REVIEW_HOUR` ET (probed every 10min by the scheduler), a single DeepSeek call reviews all holdings — stale-thesis positions are sold (`trigger: 'review'`) or get tightened stops. Review items naming non-held symbols are discarded.

### Real-time push (SSE)

`server/services/bus.js` keeps the set of SSE clients for `GET /api/stream`. Backend code calls `broadcast(event, data)` at each pipeline step; event names are `news`, `analysis`, `trade`, `portfolio`, `snapshot`, `cycle`, `reset`. Quote/valuation pushes (every `QUOTE_PUSH_SECONDS`) only run while `clientCount() > 0` to save FMP quota, and snapshots throttle to one per 30 minutes while the market is closed. The frontend (`web/src/App.jsx`) consumes these via `EventSource`: `portfolio`/`snapshot` carry full payloads applied directly to state, the other events trigger targeted re-fetches (`news`/`analysis` only bump a `newsVersion` counter — the news tab fetches its own data with server-side filter/search/pagination via `/api/news?sentiment=&q=`). The 60s full-refresh polling runs **only while SSE is down** (plus one catch-up refresh on reconnect), and long-lived client state is capped (snapshots ≤600 points, news list ≤600 items) so a dashboard left open for days doesn't grow unbounded. If you add a new event type, wire both sides.

### Prices and market sessions

`fmp.js#getQuote` is the single source of price truth: outside regular hours it merges the FMP aftermarket-trade price and exposes `effective_price` (use this, not `price`, for valuation/fills/stop checks), plus `session` (`pre`/`regular`/`post`/`closed` computed from US Eastern time) and `extended_price`. Quotes are cached ~10s and profiles 24h in in-process Maps. `getMarketSession` consults `marketCalendar.js` (rule-computed NYSE calendar, never goes stale): full holidays return `closed`, half days (Jul 3 / day after Thanksgiving / Christmas Eve) end `regular` at 13:00 ET with `post` until 17:00; `isTradingDay` from the same module filters the daily series used for Sharpe. The SPY benchmark uses `getHistoricalPricesAdjusted` (dividend-adjusted total return, falls back to the unadjusted light endpoint with `basis: 'price'`).

### Database conventions

- Supabase Postgres accessed only from the server with the `service_role` key (`server/db.js`); RLS allows public read on all tables, writes bypass RLS. The frontend never talks to Supabase directly — everything goes through `/api/*`.
- `portfolio_state` is a single-row table (`id = 1`) holding cash; positions use weighted-average cost (plus `stop_loss`/`take_profit`/`peak_price`); `trades.realized_pnl` is set on sells; `trade_reflections` stores per-closed-trade lessons (006); `pending_orders` holds closed-session signals awaiting the next open (010, statuses `pending/filled/cancelled/expired`); forward returns live on `news_analyses` (`signal_price`/`fwd_return_*`, 011). Admin reset truncates `trades/news_*` etc. with `cascade`, which also clears `trade_reflections`/`pending_orders`.
- **Migration-tolerant code:** schema changes ship as `supabase/migrations/NNN_*.sql` (also folded into `schema.sql`), and server code degrades gracefully when a migration hasn't been applied — e.g. `analyzeAndStore` retries without `event_summary`, `/api/snapshots` falls back when the `snapshots_sampled` RPC is missing. Follow this pattern when touching the schema.

### DeepSeek usage

All LLM calls go through `chatJSON` in `server/services/deepseek.js` (OpenAI-compatible chat completions with `response_format: json_object`, 90s timeout). The six prompts (analyst, trader, event-matcher, risk-officer, position-review, reflection) live in that file as constants and demand strict JSON shapes; outputs are defensively clamped/validated in code. News titles/bodies are untrusted external input: they pass through `sanitizeUntrusted` (strips control/bidi chars, truncates) and are framed with explicit ignore-embedded-instructions notes (`UNTRUSTED_NOTE`) in the analyst/trader/event-matcher prompts — keep that framing when adding prompts that quote article text. Cost is bounded by `MAX_ANALYZE_PER_CYCLE` (default 8 articles per round); the newer calls are deliberately off the hot path (risk officer: once per surviving buy signal; reflection: once per closing sell; review: once per trading day) — keep that in mind before adding LLM calls to hot loops.

## Conventions

- ESM throughout (`"type": "module"`); plain JavaScript, no TypeScript, no framework beyond Express/React.
- Comments, log messages, commit messages, and all user-facing UI copy are written in Simplified Chinese; identifiers are English. Logs are prefixed by module: `[cycle]`, `[trader]`, `[risk]`, `[fmp]`, `[event]`, `[api]`, `[scheduler]`, `[admin]`, `[memory]`, `[review]`, `[riskofficer]`, `[perf]`, `[queue]`, `[signal]`.
- The UI is an emoji-free, **light-theme** dashboard built on **Ant Design 5** (`ConfigProvider` with `zh_CN` locale + `themeConfig` in `web/src/theme.js`; charts stay Recharts) and is mobile responsive via antd Grid (`Row`/`Col` breakpoints, `Table scroll={{x}}` for wide tables). Colors follow the **US market convention: green = gain/利好/buy, red = loss/利空/sell**, encoded once in `web/src/theme.js` (`COLOR_UP`/`COLOR_DOWN`, `CHART`, `PIE_COLORS`) and the `.up`/`.down` utility classes — never hardcode PnL colors elsewhere or repurpose antd success/error tokens for it. `styles.css` only holds layout/utility helpers; toasts go through antd `App.useApp().notification` (the `AntApp` wrapper lives in `main.jsx`). Shared formatters and label maps (`TIER_LABELS`, `SESSION_LABELS`, `TRIGGER_LABELS`) live in `web/src/api.js`.
- Public-facing UI copy and public API responses must not name third-party providers or internals (FMP, DeepSeek, Yahoo, model IDs) — that information lives only in docs and the token-gated admin page (`/api/admin/status` includes the model; public `/api/status` does not).
- External calls (FMP, Yahoo, DeepSeek) use native `fetch` with `AbortSignal.timeout(...)`; per-source failures are collected and logged, never allowed to kill a whole cycle (`Promise.allSettled` in fetch, try/catch per article in analyze).
