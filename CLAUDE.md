# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

AI 新闻交易员 — a news-driven US stock paper-trading site. It fetches financial news (FMP API + Yahoo RSS), has DeepSeek classify each article as bullish/bearish on a four-tier scale, lets DeepSeek decide simulated buy/sell orders under server-side risk constraints, and shows the portfolio live in a React dashboard. Everything is virtual money; there is no real brokerage integration.

## Commands

```bash
npm install                # root deps (Express server)
npm run build              # cd web && npm install && vite build → web/dist
npm start                  # node server/index.js → http://localhost:3000 (serves API + built frontend)
npm test                   # node --test → test/*.test.js (pure-function unit tests, no env/network needed)

# Frontend dev with hot reload (two terminals):
npm start                  # terminal 1: backend on :3000
cd web && npm run dev      # terminal 2: Vite on :5173, /api proxied to :3000
```

- Requires Node >= 22. Unit tests cover the pure functions (market calendar, slippage model, tier/credibility/final-confidence math, sizing chain, eligibility gate, forward returns, signal stats); CI (`.github/workflows/ci.yml`) runs `npm test` + the frontend build on every push/PR. No linter configured. When adding pure logic, prefer extracting it into a testable module (see `sizing.js`, `eligibility.js`) and add a test.
- `.env` (copy from `.env.example`) needs `FMP_API_KEY`, `DEEPSEEK_API_KEY`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`. The server still boots without them (API + static hosting work) but the scheduler does not start. All tunables live in `server/config.js`; document new env vars in `.env.example`, `README.md`, and `render.yaml`.
- Database: run `supabase/schema.sql` in the Supabase SQL Editor for a fresh install; existing deployments apply incremental scripts from `supabase/migrations/` instead.
- Deployed on Render via `render.yaml` (build = `npm install && npm run build`, start = `npm start`, health check = `/api/health`).
- Manual trigger for a full cycle: `POST /api/run-cycle` (requires `x-admin-token` header if `ADMIN_TOKEN` is set; anonymous triggers share a global 120s cooldown and get 429 inside it). The endpoint exists for external cron services — the public dashboard has no trigger button; in-site manual runs go through `#/admin`.
- Admin surface: the hidden frontend route `#/admin` plus `/api/admin/*` (`server/routes/admin.js`) — verify/status/metrics/trading-halt/run-cycle/reset. These endpoints **require** `ADMIN_TOKEN` (503 when unset, 403 on mismatch); token comparison is constant-time (`authGuard.js#safeTokenEqual`) and failed attempts are rate-limited per IP (10 per 15min via `express-rate-limit`, successes don't count) on both `/api/admin/*` and `/api/run-cycle`. `POST /api/admin/reset` wipes all business data and restores cash to `INITIAL_CAPITAL`: it sets the global halt flag (`server/services/halt.js`), drains the running cycle, then truncates via the `admin_reset_data` RPC (005 migration; per-table delete fallback) inside the trade lock, clears in-process caches, and broadcasts a `reset` SSE event.

## Architecture

Single Node process: Express serves `/api/*` and the built frontend from `web/dist`, and `server/scheduler.js` runs twelve `setInterval` loops (news poll, quote push, snapshot, risk check, daily position review, open-order queue, signal forward-return backfill, plus — when `ENABLE_MACRO` — economic-calendar refresh, macro-regime decay recompute, the capital-allocator tick, the deterministic market-check refresh when `ENABLE_MARKET_CHECK`, and — when `ENABLE_SHADOW` — the shadow-portfolio stop check). There is no queue or worker — all background work happens in-process. Every scheduler loop is wrapped by `singleton.js#makeSingleton` (a tick that fires while the previous run is still going is skipped, with a rate-limited stuck-task warning after 10min) — the modules' own re-entrancy flags (`cycleStatus.running`, `running` in riskMonitor/positionReview/openQueue/signalReturns, `allocatorStatus.running`) remain as a second layer because they also guard manual/API entry points — plus the global halt flag in `halt.js` (set during admin reset; runCycle, checkStops, processPendingOrders, backfillForwardReturns and runAllocation early-return while halted; admin reset drains both the running cycle and the running allocation before truncating).

### The trading pipeline (core flow)

`runCycle()` in `server/services/newsService.js` orchestrates one round and is the place to start reading:

1. **Fetch** — `fmp.js` (stock/general/press news) + `yahoo.js` (RSS for watchlist + held symbols). Fast polls (every `NEWS_POLL_SECONDS`, default 20s) only fetch stock news; roughly every 5 minutes a `fullFetch` round adds the other sources. Articles are upserted into `news_articles` with `onConflict: 'url', ignoreDuplicates: true`, so only genuinely new rows flow onward. At normalization time each article gets a **source credibility score** (`credibility.js#scoreSource`, stored in `source_domain`/`source_score`, 009 migration): the *original* source is scored by article-URL domain (Reuters/AP/SEC 0.95 → press wires 0.90 → Bloomberg/WSJ/CNBC/Yahoo 0.85 → opinion platforms 0.65 → low-trust sites 0.50 → unknown 0.40), minus a 0.03 aggregation penalty for FMP-relayed articles — FMP/Yahoo are discovery channels, not the scored source.
2. **Analyze** — candidates come from a backlog query (articles from the last 24h with `analyzed_at IS NULL`), not just the current round's inserts, capped at `MAX_ANALYZE_PER_CYCLE` per round; `analyzed_at` is stamped after each analysis (including irrelevant verdicts) so failures retry next round. `deepseek.js#analyzeArticle` returns sentiment, impact strength/scope, confidence, and a one-line `event_summary`. Tier is computed in code by `computeTier` (1 = strong+wide … 4 = weak+narrow), not by the LLM. `analyzeAndStore` also computes `final_confidence` = source_score × LLM confidence × recency (1.0 within 1h, floor 0.5 at 24h+) × tier materiality (`credibility.js#computeFinalConfidence`). Every non-neutral tiered analysis also gets its **signal-time market price** stamped (`signalReturns.js#recordSignalPrice` → `news_analyses.signal_price`, 011 migration) — including signals later deduped/held/vetoed — feeding the evaluation layer below. A signal is actionable only if non-neutral, `tier <= TRADE_TIER_THRESHOLD` (default 2), and confidence ≥ 0.5.
3. **Dedup + cross-source confirmation** — `eventService.js#resolveEvent`: DeepSeek compares the event summary against the symbol's `news_events` rows from the last `EVENT_DEDUP_HOURS` (72h); duplicate coverage of the same underlying event only bumps `article_count` and never trades again. At the threshold comparison, bullish signals from press-release sources (`credibility.js#isPressRelease`: `fmp-press` or newswire domains) are discounted by `PRESS_BULLISH_PENALTY` (default ×0.75) — press releases are factual but inherently promotional and the classic micro-cap pump channel, so a lone PR usually lands in the held bucket until an independent outlet confirms. Signals whose (penalized) `final_confidence < MIN_FINAL_CONFIDENCE` (default 0.35) are **held**: the event is recorded but not traded. If a later report of a still-untraded event arrives from an *independent* domain (tracked in `news_events.source_domains`), `resolveEvent` returns `confirmable: true` and `runCycle` re-evaluates with the new article's own final_confidence × confirmation boost (+10% per extra source, cap ×1.2) — a lone low-trust site can't trigger a trade, but a Reuters/Yahoo follow-up confirms it. New/confirmed events still pass a same-direction trade cooldown (`TRADE_COOLDOWN_MINUTES`, 30min) on the `trades` table. **Fail-closed convention:** if dedup checks error out, the trade is skipped — prefer missing a trade over double-trading (likewise, no cross-confirmation when `source_domains` is missing).
4. **Pool / sell** — `trader.js#handleSignal` fetches quote + company profile (`isActivelyTrading === false` skipped). Bullish signals must first pass the **eligibility gate** (`eligibility.js#checkBuyEligibility`, before any LLM call): exchange whitelist (`ALLOWED_EXCHANGES`, default NASDAQ/NYSE/AMEX — auto-blocks OTC/pink sheets; AMEX aliases normalized via `normalizeExchange`), ETF/fund filter (`isEtf`/`isFund === true` rejected; missing fields pass — the one deliberate fail-open exception, normal stocks often lack these fields), min market cap (`MIN_MARKET_CAP`, $300M), min share price (`MIN_PRICE`, $2), min average dollar volume (`MIN_AVG_DOLLAR_VOLUME`, $5M/day); missing profile data otherwise fails closed. Eligible bullish signals are then **pooled, not traded** (014): `poolBullishSignal` inserts a `candidate_signals` row (base score, sector, macro-regime snapshot, pool-entry market price `entry_price` for queueing-cost measurement (016); born as `conflict_hold` if the symbol has a pending sell order) and returns `{ pooled: true }` — **no LLM trade call happens at signal time**; runCycle marks the event consumed (`summary.pooled`). If the pool is unavailable (014 not applied / `ENABLE_MACRO=false`), the original immediate path below runs unchanged. Bearish signals first freeze same-symbol buy candidates (`candidateStore.js#holdBuyCandidates` → `conflict_hold`, fail-open) and then proceed through the immediate LLM sell flow as before (sells are never pooled or budget-gated). The immediate/legacy buy path and the allocation-time path share the same machinery: `deepseek.js#decideTrade` must first pass *symbol validation* (news subject really this listed company? price/market-cap magnitudes match?) — guards against mapping pre-IPO companies to similar tickers (SpaceX ≠ SPCE); validation failure forces hold. Sizing chain (in order): LLM `fraction` (**a share of total portfolio value**, capped by available cash — not a share of remaining cash, which would make position size depend on news arrival order) → tier/confidence/source-credibility multipliers (`sizing.js#scaleFraction`) → macro×sector×conflict scaling (allocation path only) → consecutive-loss size-down → risk-officer `scale` → hard caps. The **risk officer** (`deepseek.js#reviewProposedTrade`, `ENABLE_RISK_OFFICER`) reviews each surviving buy with portfolio context + macro block and can approve/downsize/veto; only explicit `approve === true` passes, call failure skips the buy (fail-closed). **Portfolio-level hard risk stops** (013/014, `riskControls.js` + `tradingHalt.js` + regime params, all code-enforced inside `settleBuyLocked` within the trade lock, buys only — sells/stops always pass): admin kill switch (`trading_halted` on `portfolio_state`, persisted, toggled via `POST /api/admin/trading-halt`), daily loss circuit breaker (`DAILY_LOSS_HALT_PERCENT`, sticky per ET day, fail-open with one warn/day when no baseline), max open positions (`MAX_OPEN_POSITIONS`, adds exempt), `macro_shock` full buy stop, daily new-position quota (`MAX_NEW_POSITIONS_PER_DAY`, adds exempt, reseeded from today's buys on restart — conservative overcount), the **regime triple clamp** (`computeBuyHeadroom`: cash reserve floor / daily buy budget vs day-start baseline / gross exposure cap, per `config.macroRegimeParams`), sector concentration clamp (`MAX_SECTOR_FRACTION`), and hard caps (single position ≤ 25%, single buy ≤ 20%, min order $50, long-only). All such rejections carry a `reason` code; temporary ones (`trading_halted`/`daily_loss_halt`/`macro_shock`/budget-class) return `{ reject, transient: true }` — `openQueue.js` keeps pending orders and the allocator keeps candidates for retry instead of cancelling. Buys store AI-chosen `stop_loss`/`take_profit` (clamped 3–15% / 5–30%); fills go through the trade lock (`withTradeLock`), re-fetch valuation + fresh quote (drift abort `BUY_PRICE_DRIFT_ABORT_PERCENT`, default 5%), the slippage model (`execution.js#computeFill` — both trader call sites pass `minutesSinceOpen` from `marketCalendar.js#minutesSinceMarketOpen`, so fills in the first `OPENING_WINDOW_MINUTES` (15) of the regular session get an extra ×`OPENING_SLIPPAGE_MULT` (2) — the open-trigger allocation run and open-queue fills land in the widest-spread window) and the atomic `execute_trade` RPC, with `trades.quote_price`/`slippage_bps`/`macro_regime` recorded best-effort post-insert; pool-path fills also record `pool_entry_price`/`pool_wait_minutes`/`pool_drift_percent` (016, `execution.js#computePoolMetrics`) — the measured queueing cost of the candidate pool. **Open-order queue (010):** closed-session SELLS (and legacy-path buys) are enqueued into `pending_orders` and filled at the next open by `openQueue.js#processPendingOrders`; closed-session bullish signals now land in the candidate pool instead, which the allocator clears at the open. Pre/post sessions still fill immediately at real aftermarket prices.
5. **Allocate** (014, the user-specified cadence) — `allocator.js#maybeRunAllocation` (scheduler tick every 60s) runs `runAllocation` only in the regular session: the first tick after the open executes immediately (clearing the overnight pool), then every `ALLOCATION_INTERVAL_MINUTES` (default 15) — so intraday news-to-trade latency is ≤15min by design, and off-hours candidates only accumulate and get re-scored. Each run: expire stale candidates (`CANDIDATE_MAX_AGE_HOURS`, 24h) → `macro_shock` gate (cancel low-score candidates, keep the rest, no buys) → economic-calendar **blackout** gate (`macroCalendar.js#isInBlackout`, ±30min around high-importance US releases; fail-open if the FMP plan lacks the endpoint) → refresh scores (`scoreCandidate` = tierScore × LLM confidence × recency decay × source score × regime `macroMultiplier` × `sectorMultiplier`; writes are batched — `shouldWriteScore` skips score-only changes < 0.01, the rest flush concurrently in chunks of 20) → regime tier/confidence filter (`allowedTiers`, risk_off also needs conf ≥0.7; filtered candidates marked `macro_filtered`, non-terminal) → **conflict resolution** (`conflictResolver.js#resolveConflicts`, pure: strength = final_confidence × tierScore, 1.5× dominance ratio; pending sell order → hold, bearish dominant & held → cancel, comparable → hold, bullish dominant → half-size unless risk_off; re-judged every run so holds auto-release when the opposing signal ages out of `CONFLICT_WINDOW_MINUTES`) → `mergeBySymbol` (best per symbol + small multi-event bonus) → rank → execute top `MAX_ALLOCATIONS_PER_RUN` (default 3) via `trader.js#executeCandidate` (re-checks eligibility + trade cooldown via `eventService.js#checkTradeCooldown`, then decideTrade **with macro context** → sizing → officer → settle). LLM cost only happens here, for the top-ranked few. **Concurrency (016):** the allocator's candidate writes use optimistic concurrency (`candidateStore.js#updateCandidate` with `expectedStatus` — a write that misses means another path, typically `holdBuyCandidates` from a fresh bearish signal, changed the row first and wins) and each candidate's status is re-read (`getCandidateStatus`) right before `executeCandidate` — closing the read-snapshot→LLM-call→execute race; only the post-trade `allocated` write is unconditional (the trade already happened). Result mapping: trade → `allocated` (+ sibling candidates cancelled); budget-class reasons (`cash_reserve`/`daily_budget`/`gross_exposure`/`new_position_quota`) → this and all lower-ranked candidates `capital_constrained` (stay in pool, re-evaluated when cash frees up); global transients stop the run; permanent rejects → `rejected`. Candidate statuses: `pending/capital_constrained/macro_filtered/conflict_hold` are re-evaluable, `allocated/rejected/expired/cancelled` terminal.

Separately, `riskMonitor.js#checkStops` (every `RISK_CHECK_SECONDS`, skipped while market is closed) sells the full position when price crosses stop-loss/take-profit, reusing `trader.js#executeSellOrder` with `trigger: 'stop_loss' | 'take_profit'` (news trades use `trigger: 'news'`, daily review sells use `trigger: 'review'`). It also ratchets trailing stops (`ENABLE_TRAILING_STOP`, needs the 007 `peak_price` column): on a new post-entry high, `stop_loss` moves up to `peak × (1 − original stop distance)`, never down.

### Macro signal layer (014)

Symbol-less `fmp-general` articles are routed (in the runCycle analyze loop, before the stock analyst) to `macroService.js#processMacroArticle` → `deepseek.js#analyzeMacroArticle` (purpose `macro-analyst`, same sanitizeUntrusted/UNTRUSTED_NOTE framing): event type (CPI/FOMC/NFP/yields/tariffs/geopolitics/…), risk_on/risk_off direction, rates/inflation/growth signals, affected sectors (FMP 11-sector whitelist), impact tier 1–3, confidence — stored in `macro_events` (numeric surprise backfilled from the economic calendar when a same-day matching release exists). **Macro event dedup (015):** before insert, the new result is LLM-matched against the last 72h of `macro_events` (`deepseek.js#matchMacroEvent`, purpose `macro-event-matcher`, candidates *not* filtered by event_type — the same underlying event can be classified differently); a duplicate report only bumps `article_count` and nudges `confidence` up (`macroRegime.js#mergeMacroConfidence`: max × 1.05, capped 0.95) on the original row — `created_at` stays at first-report time, so duplicates never refresh decay/validity ("续命" happens only via genuinely new developments the LLM classifies as new events). Dedup failure is **fail-open** (insert as before — opposite of the stock-side convention, because dropping a macro event means missing a real risk signal, while a stray duplicate only skews the score); missing 015 columns degrade to dedup-without-counting via strip-and-retry. The **regime** is aggregated entirely in code (`macroRegime.js#aggregateRegime`, pure + tested): events within `MACRO_EVENT_VALIDITY_HOURS` (72h) weighted by tier (1.0/0.6/0.3) × confidence × **source score** (016: `macro_events` rows carry `source_domain`/`source_score`/`source_domains`; clamp 0.4–1, 0.7 default for pre-016 rows — a low-trust site's headline no longer weighs like Reuters) × exp(−age/24h), smoothed into a [-1,1] risk score → `risk_on/neutral/risk_off` with hysteresis (enter ±0.30 / stay ±0.20); as a deterministic backstop for dedup misses, same `event_type`+`macro_direction` events are geometrically dampened within the risk-score sum (heaviest first, ×0.6 per extra row) while distinct types keep full weight. A tier-1 ≥0.75-confidence risk_off event hard-triggers `macro_shock` for `MACRO_SHOCK_HOURS` (6h) **only with corroboration** (016, `shockCorroborated`): merged `article_count ≥ MACRO_SHOCK_MIN_REPORTS` (2) or ≥2 distinct `source_domains` (dedup merges accumulate domains and keep the max source_score); rows lacking both fields (pre-015/016) fall back to the old single-report trigger — the safety line must not silently die on a missing migration. State persists in single-row `macro_state` (survives restarts, reloaded by `initMacroRegime` at startup), is recomputed after each macro event and by a 10-min decay loop, and broadcasts a `macro` SSE event on switch. `getRegime()` is synchronous (in-process cache) so `settleBuyLocked` can consult it inside the lock; per-regime params live in `config.macroRegimeParams` (cash reserve / daily budget / gross exposure / buy multiplier / allowed tiers). **Deterministic market check (016, `marketCheck.js`, log prefix `[market]`):** a scheduler loop (`MARKET_CHECK_POLL_MINUTES`, 10min) classifies SPY vs its 20-day SMA (±0.5% buffer) + VIX level (<20 risk_on / ≥26 risk_off) into a news-independent trend; `macroRegime.js#getEffectiveRegime()` is the **single choke point** where it intersects the news regime (`intersectRegime`, pure: only news `risk_on` with a non-agreeing available check is clamped to neutral params — never loosens risk_off/macro_shock; unavailable check passes through). The allocator, `settleBuyLocked` and `/api/macro` all take params from `getEffectiveRegime()`; `regime.regime` stays the news regime (shock gate / fill snapshots unchanged). VIX quote unavailable (plan 403/404) degrades once to SPY-only; SPY data unavailable disables the check entirely (fail-open). `macroCalendar.js` polls FMP `/economic-calendar` (every `CALENDAR_POLL_MINUTES`); a 403/404 marks the endpoint unavailable once and disables only the blackout/surprise features (fail-open — the macro layer is risk-tightening, but blackout is an enhancement, not a safety line). Everything in this layer is migration-tolerant and must never break the stock-trading path: missing 014 tables disable the pool/regime (regime stays `neutral`) and trading falls back to the 013 behavior.

### Observability (012)

Each `runCycle` gets a `run_id` (UUID) and persists per-run metrics to `cycle_runs` (counts, duration, LLM calls/tokens/estimated cost, provider error counts, reject-reason distribution, full error texts) via `cycleRuns.js#saveCycleRun` in the `finally` block — pure observation layer, **fail-open**: any metrics/persistence failure must never break the trading path. `metrics.js` holds the in-process state: per-ET-day LLM usage (by purpose) + provider error counters (incremented in `deepseek.js#chatJSONWithMeta` and `fmp.js#fmpGet`), and a module-level current-run accumulator (`beginRun`/`endRun`/`currentRunId`/`recordReject`) — runCycle is single-flight so no ctx threading is needed; concurrent background LLM calls (reflection/review/queue fills) get attributed to whatever run is open, an accepted approximation. `news_analyses` rows carry `run_id`/`llm_latency_ms`/`llm_*_tokens`; `trades` rows carry `run_id`/`decision_started_at`/`decision_finished_at` (the decideTrade→risk-officer window whose staleness the drift abort guards) and `quote_timestamp` (the fill quote's own FMP timestamp), written via the same best-effort post-insert update as 008 (`recordFillDetails`, per-column strip-and-retry). Queued fills and stop/review sells have no `run_id` (no run context) — that's expected. `cycle_runs` has **no public-read RLS policy** (exception to the convention: its `errors` contain provider names, which are only allowed on the token-gated admin surface); the admin page reads it via `GET /api/admin/metrics` (recent runs, today's LLM cost, backlog/pending counts, reject reasons). Public-facing cycle errors (`/api/status` lastResult/lastError, SSE `cycle` event) are sanitized through `metrics.js#sanitizeProviderText`. **Decision replay (018, `decisionLog.js`, log prefix `[decision]`, `ENABLE_DECISION_LOG`):** every `decideTrade` episode (immediate + allocation paths) writes one `trade_decisions` row — prompt version (`deepseek.js#PROMPT_VERSIONS`, **bump when editing the trader/risk-officer system prompt text**), the full messages array (replayable as-is), a sha256 input hash, the raw LLM JSON and the normalized decision for both trader and officer, the sizing-chain steps, decision/fill price snapshots and a terminal outcome (executed/queued/hold/symbol_invalid/vetoed/officer_error/rejected/sell_skipped). `decideTrade`/`reviewProposedTrade` attach a `replay` property to their returns; `trader.js` calls `beginDecisionEpisode`/`attachOfficer`/`finishDecision` at every exit point. Fail-open, missing-018 disables it; like `cycle_runs`, the table has **no public-read RLS policy** (full prompts embed portfolio details) and is exposed only via token-gated `GET /api/admin/decisions` (`?full=1` for messages/raw).

### Signal evaluation layer (forward returns)

Independent of trading, `signalReturns.js` answers "does the classification itself have alpha": every non-neutral analysis gets `signal_price` at observation time, and a 10-minute scheduler loop backfills `fwd_return_1h` (live quote in the t+60–120min window; left null if missed) and `fwd_return_1d`/`fwd_return_5d` (close of the 1st/5th trading day after the signal's ET date, via daily history — `computeDailyForwardReturns` is pure and tested). `signalStats.js#getSignalStats` (`/api/signal-stats`, "信号质量" tab) aggregates direction-adjusted hit rates / mean returns bucketed by tier, source-score band, confidence (calibration), traded-vs-untraded, **macro regime at pool entry** (`candidate_signals.macro_regime` — the evidence chain for whether the macro layer adds value) and **execution path / queue-wait** (immediate vs pooled, wait buckets from `trades.pool_wait_minutes`, 016 — quantifies the alpha cost of pooling delay; a top-level `pooling` aggregate reports avg wait + avg pool→fill drift), plus a final_confidence↔return Pearson IC per horizon. Hit rates carry a **Wilson 95% CI** (`wilsonInterval`, pure) and the UI colors them only when the interval clears 50% — small samples stay neutral. `getSignalStats({ days })` supports a time window (`?days=`, UI presets 7d/30d/all) with paginated fetches past the PostgREST 1000-row cap (truncation surfaced via `window.truncated`/`covered_since`), and the traded-vs-blocked group is an explicit **opportunity-cost** breakdown (capital_constrained / macro_filtered / conflict_hold / allocation-path officer vetoes detected from `candidate_signals.status_reason` prefix / expired) — blocked signals with persistently positive forward returns mean the layer is over-conservative. All of it is migration-tolerant (011/016) and never touches the trading path. **Parameter advisor** (`parameterAdvisor.js`, `GET /api/admin/advisor`, admin page card): deterministic rules (no LLM) map these stats back to parameter suggestions — low-credibility bullish losses → raise `MIN_FINAL_CONFIDENCE`, press-bullish hit rate CI below 50% → deepen `PRESS_BULLISH_PENALTY`, tier-2 beating tier-1 on 5d with non-overlapping CIs → recalibrate tiers, blocked-signal opportunity cost per layer (adjust when blocked mean ≥ +0.3%, confirm value when ≤ −0.3%), confidence-calibration inversion, plus shadow-ablation cross-checks (variant vs actual window return, ≥14 days, ±2% edge). Each rule has a minimum-sample gate and emits nothing when insignificant (`evaluateSignalRules`/`evaluateShadowRules` are pure and tested; thresholds in `ADVISOR_THRESHOLDS`); it only suggests — parameters are still changed by humans via env vars.

Two post-trade LLM features (both cheap by design):
- **Reflection/memory** (`memoryService.js`, `ENABLE_REFLECTION`, 006 migration): after any sell with `realized_pnl`, `executeSellOrder` fires `reflectOnClosedTrade` outside the trade lock (fire-and-forget) — DeepSeek distills a transferable lesson into `trade_reflections`; `getMemories` injects ≤5 lessons into later `decideTrade`/risk-officer calls. The prompt judges **alpha, not beta** (016): `benchmark.js#getHoldingBenchmark` supplies the holding-period SPY return (same-ET-day holds ≈ SPY day change, multi-day = dividend-adjusted close-to-close via the pure `spyHoldingReturn`) and the excess return, stored in `trade_reflections.spy_return_percent`/`excess_return_percent`; benchmark fetch failure just omits the field (fail-open).
- **Daily position review** (`positionReview.js`, `ENABLE_POSITION_REVIEW`): once per trading day after `POSITION_REVIEW_HOUR` ET (probed every 10min by the scheduler), a single DeepSeek call reviews all holdings — stale-thesis positions are sold (`trigger: 'review'`) or get tightened stops; each position carries its holding-period SPY return (016, same benchmark helpers) so the model judges performance relative to the market. Review items naming non-held symbols are discarded.

### Shadow portfolios / ablation (017)

`shadowPortfolio.js` (orchestration, log prefix `[shadow]`) + `shadowEngine.js` (pure math, tested) run several virtual portfolios in parallel with the real one, each with one defense layer turned off, to answer "which layer actually adds return vs merely reduces trading": `no_risk_officer` (mirrors actual fills un-applying the officer's scale via `effectiveFraction ÷ officerScale`, plus executes officer-vetoed buys at the pre-veto plan — hooks at both veto/error sites in `handleSignal`/`executeCandidate`), `no_macro_filter` (mirrors at the pre-macro-clamp `requestFraction`, plus replays buys blocked by the regime tier filter / `macro_shock` / blackout (allocator hooks, top `MAX_ALLOCATIONS_PER_RUN` by score) and by the settle-time macro rejects (`macro_shock`/`new_position_quota`/triple-clamp bindings)), `immediate_trade` and `equal_weight` (independent strategies fed by `onBullishSignal`/`onBearishSignal` in `handleSignal` after the eligibility gate: buy at signal time with deterministic sizing — `SHADOW_BASE_FRACTION` × the sizing.js chain, or the flat `SHADOW_EQUAL_WEIGHT_FRACTION` — sell-all on bearish; they never mirror actual trades), plus `spy_benchmark` (one-shot all-in SPY) and `cash`. Key invariants: **zero extra LLM calls** (paths the real system never sent to the LLM use deterministic sizing + default stops — an ablation approximation, not a replay); **fail-open observation layer** (every hook is fire-and-forget through an in-process serial queue; missing 017 tables disable the module with one warning); **one buy per (variant, analysis_id)** (unique partial index, 019 — DB-level hard constraint, the in-process check-then-insert is just the first line; prevents re-buys when macro-filtered candidates replay every allocation run or a pooled candidate later fills for real and gets mirrored). Fills reuse `execution.js#computeFill` (mirrors take the actual fill price); shadow positions get their own stop/take monitoring (`checkShadowStops`, same cadence as riskMonitor, quotes mostly served from cache) and 10-min-throttled snapshots piggybacking the snapshot loop. `GET /api/shadow` (前端 "消融实验" tab) returns per-variant valuations, window-rebased equity series (`shadow_snapshots_sampled` RPC with plain-query fallback) and recent shadow trades, alongside the actual portfolio for the same window. Admin reset truncates the shadow tables and re-inits the variants.

### Real-time push (SSE)

`server/services/bus.js` keeps the set of SSE clients for `GET /api/stream`. Backend code calls `broadcast(event, data)` at each pipeline step; event names are `news`, `analysis`, `trade`, `portfolio`, `snapshot`, `cycle`, `reset`, `macro` (regime switches + pool-count updates; the frontend bumps a `macroVersion` counter and the 宏观 tab refetches `/api/macro`). Quote/valuation pushes (every `QUOTE_PUSH_SECONDS`) only run while `clientCount() > 0` to save FMP quota, and snapshots throttle to one per 30 minutes while the market is closed. The frontend (`web/src/App.jsx`) consumes these via `EventSource`: `portfolio`/`snapshot` carry full payloads applied directly to state, the other events trigger targeted re-fetches (`news`/`analysis` only bump a `newsVersion` counter — the news tab fetches its own data with server-side filter/search/pagination via `/api/news?sentiment=&q=`). The 60s full-refresh polling runs **only while SSE is down** (plus one catch-up refresh on reconnect), and long-lived client state is capped (snapshots ≤600 points, news list ≤600 items) so a dashboard left open for days doesn't grow unbounded. If you add a new event type, wire both sides.

### Prices and market sessions

`fmp.js#getQuote` is the single source of price truth: outside regular hours it merges the FMP aftermarket-trade price and exposes `effective_price` (use this, not `price`, for valuation/fills/stop checks), plus `session` (`pre`/`regular`/`post`/`closed` computed from US Eastern time) and `extended_price`. Quotes are cached ~10s and profiles 24h in in-process Maps. `getMarketSession` consults `marketCalendar.js` (rule-computed NYSE calendar, never goes stale): full holidays return `closed`, half days (Jul 3 / day after Thanksgiving / Christmas Eve) end `regular` at 13:00 ET with `post` until 17:00; `isTradingDay` from the same module filters the daily series used for Sharpe. The SPY benchmark uses `getHistoricalPricesAdjusted` (dividend-adjusted total return, falls back to the unadjusted light endpoint with `basis: 'price'`).

### Database conventions

- Supabase Postgres accessed only from the server with the `service_role` key (`server/db.js`); RLS allows public read on all tables, writes bypass RLS. The frontend never talks to Supabase directly — everything goes through `/api/*`.
- `portfolio_state` is a single-row table (`id = 1`) holding cash; positions use weighted-average cost (plus `stop_loss`/`take_profit`/`peak_price`); `trades.realized_pnl` is set on sells; `trade_reflections` stores per-closed-trade lessons (006); `pending_orders` holds closed-session signals awaiting the next open (010, statuses `pending/filled/cancelled/expired`); forward returns live on `news_analyses` (`signal_price`/`fwd_return_*`, 011); `macro_events`/`macro_state`(single-row)/`candidate_signals` hold the macro layer and buy-candidate pool, and `trades.macro_regime` snapshots the regime at fill time (014); 016 adds `candidate_signals.entry_price`, `trades.pool_*` (queueing cost), `macro_events.source_domain/source_score/source_domains` (credibility + shock corroboration) and `trade_reflections.spy_return_percent/excess_return_percent` (alpha benchmark). 017 adds the shadow-portfolio tables (`shadow_portfolios`/`shadow_positions`/`shadow_trades`/`shadow_snapshots`, public-read RLS). 018 adds `trade_decisions` (decision replay, RLS enabled but no public-read policy). 019 upgrades the shadow buy-dedup index to a unique partial index. Admin reset truncates `trades/news_*/macro_events/candidate_signals/shadow_*/trade_decisions` etc. with `cascade` (also clearing `trade_reflections`/`pending_orders`), resets `macro_state` to neutral and re-initializes the shadow variants.
- **Migration-tolerant code:** schema changes ship as `supabase/migrations/NNN_*.sql` (also folded into `schema.sql`), and server code degrades gracefully when a migration hasn't been applied — e.g. `analyzeAndStore` retries without `event_summary`, `/api/snapshots` falls back when the `snapshots_sampled` RPC is missing. Follow this pattern when touching the schema.

### DeepSeek usage

All LLM calls go through `chatJSON`/`chatJSONWithMeta` in `server/services/deepseek.js` (OpenAI-compatible chat completions with `response_format: json_object`, 90s timeout; every call carries a `purpose` tag and reports latency/token usage to `metrics.js`). The seven prompts (analyst, macro-analyst, trader, event-matcher, risk-officer, position-review, reflection) live in that file as constants and demand strict JSON shapes; outputs are defensively clamped/validated in code. News titles/bodies are untrusted external input: they pass through `sanitizeUntrusted` (strips control/bidi chars, truncates) and are framed with explicit ignore-embedded-instructions notes (`UNTRUSTED_NOTE`) in the analyst/macro-analyst/trader/event-matcher prompts — keep that framing when adding prompts that quote article text. Cost is bounded by `MAX_ANALYZE_PER_CYCLE` (default 8 articles per round); the newer calls are deliberately off the hot path (risk officer: once per surviving buy; reflection: once per closing sell; review: once per trading day; macro-analyst: once per symbol-less general article within the same MAX_ANALYZE_PER_CYCLE budget; trader+officer now run only for the allocator's top candidates — net cheaper than per-signal immediate decisions) — keep that in mind before adding LLM calls to hot loops.

## Conventions

- ESM throughout (`"type": "module"`); plain JavaScript, no TypeScript, no framework beyond Express/React.
- Comments, log messages, commit messages, and all user-facing UI copy are written in Simplified Chinese; identifiers are English. Logs are prefixed by module: `[cycle]`, `[trader]`, `[risk]`, `[fmp]`, `[event]`, `[api]`, `[scheduler]`, `[admin]`, `[memory]`, `[review]`, `[riskofficer]`, `[perf]`, `[queue]`, `[signal]`, `[metrics]`, `[macro]`, `[calendar]`, `[pool]`, `[allocator]`, `[market]`, `[shadow]`, `[decision]`, `[advisor]`.
- The UI is an emoji-free, **light-theme** dashboard built on **Ant Design 5** (`ConfigProvider` with `zh_CN` locale + `themeConfig` in `web/src/theme.js`; charts stay Recharts) and is mobile responsive via antd Grid (`Row`/`Col` breakpoints, `Table scroll={{x}}` for wide tables). Colors follow the **US market convention: green = gain/利好/buy, red = loss/利空/sell**, encoded once in `web/src/theme.js` (`COLOR_UP`/`COLOR_DOWN`, `CHART`, `PIE_COLORS`) and the `.up`/`.down` utility classes — never hardcode PnL colors elsewhere or repurpose antd success/error tokens for it. `styles.css` only holds layout/utility helpers; toasts go through antd `App.useApp().notification` (the `AntApp` wrapper lives in `main.jsx`). Shared formatters and label maps (`TIER_LABELS`, `SESSION_LABELS`, `TRIGGER_LABELS`, `REGIME_LABELS`, `CANDIDATE_STATUS_LABELS`) live in `web/src/api.js`.
- Public-facing UI copy and public API responses must not name third-party providers or internals (FMP, DeepSeek, Yahoo, model IDs) — that information lives only in docs and the token-gated admin page (`/api/admin/status` includes the model; public `/api/status` does not).
- External calls (FMP, Yahoo, DeepSeek) use native `fetch` with `AbortSignal.timeout(...)`; per-source failures are collected and logged, never allowed to kill a whole cycle (`Promise.allSettled` in fetch, try/catch per article in analyze).
